# 质量检查规范（单元测试、Pyright、Ruff）

本文档用于补充 `docstring_spec_cn.md`，定义代码变更必须满足的质量检查要求。
目标是统一单元测试、类型检查与代码风格检查，确保变更可验证、可维护、可评审。

---

## 1. 适用范围

本规范适用于仓库内所有 Python 代码变更，重点包括 `src/` 与 `demos/`。
同样适用于人工开发者与 AI Agents（如代码助手、自动化修复代理）。

必需质量门禁：
- 单元测试（`pytest`）
- 类型检查（`pyright` 严格模式）
- 代码规范检查（`ruff`）

---

## 1.1 单人开发与轻协作模式

当前仓库以个人开发为主、偶发协作为辅，执行策略如下：
- 允许简化评审流程，但不降低质量门禁。
- 即使无外部评审，提交前仍必须跑完第 2 节命令。
- 对“小功能/小修复”采用小步提交，避免大批量不可回滚改动。
- 每次提交在说明中写明变更范围与验证结果（测试、类型、lint）。

---

## 1.2 AI Agent 适配原则

AI Agent 提交必须满足与人工提交完全一致的质量标准，并额外满足可追溯性：
- 变更说明必须包含：修改文件列表、行为变化摘要、验证命令与结果。
- 不允许“仅改样式不验证”。
- 不允许未经说明新增忽略项（如 `# type: ignore`、Ruff ignore）。
- 涉及公共 API、类型收紧、行为变更时，必须同步更新测试与文档。

---

## 2. 必跑命令

在提交或合并前，必须执行以下命令：

```bash
pytest
pyright
ruff check src demos
```

建议用于自动修复 lint 问题的命令：

```bash
ruff check src demos --fix
```

若变更涉及包级导出或导入暴露，还应重新生成 stub：

```bash
python scripts/generate_init_pyi.py
```

---

## 3. 单元测试规范（pytest）

### 3.1 测试位置与命名

- 测试文件放在 `tests/` 目录。
- 文件名必须符合 `test_*.py`。
- 测试函数名应体现行为，例如 `test_builder_rejects_invalid_age_structure`。

### 3.2 测试设计要求

- 优先编写面向行为的测试，验证外部可观察结果。
- 测试必须可重复、确定性；涉及随机数时必须固定种子。
- 推荐使用 Arrange-Act-Assert 结构增强可读性。
- 单个测试应聚焦一个行为点（或一个紧密相关的断言组）。

### 3.3 Fixture 与共享初始化

- 优先复用 `tests/conftest.py` 中的 fixture，避免重复初始化逻辑。
- fixture 保持小而专注，职责清晰。
- 避免通过过深 fixture 链隐藏关键测试前提。

### 3.4 每次变更至少覆盖内容

非琐碎改动必须新增或更新测试，覆盖：
- 正常路径（happy path）
- 边界条件
- 非法输入与错误路径（适用时）
- 当前修复问题的回归场景（适用时）

### 3.5 禁止行为

- 无明确理由不得跳过测试。
- 不得保留临时调试断言或仅依赖 `print` 验证。
- 可自动化验证的功能，不得只做手工验证就合并。

---

## 4. 类型检查规范（Pyright）

### 4.1 严格模式要求

Pyright 必须满足仓库当前配置：
- `typeCheckingMode = "strict"`
- `include = ["src"]`
- `ignore = ["tests/**", "**/__pycache__", "**/*.pyc"]`

### 4.2 类型标注要求

- 对公共 API、函数签名和关键内部变量补充明确类型注解。
- 新增或修改代码中避免引入隐式 `Any`。
- 在可行时使用更精确类型（如 `Sequence[int]`、`Mapping[str, float]`、`NDArray[...]`），避免过宽泛类型。

### 4.3 错误处理策略

- 优先修复根因，而不是局部抑制报错。
- 仅在运行时不变量明确但类型系统难以表达时使用 `cast(...)`。
- `# type: ignore` 仅在最后手段使用，并附带简短且具体的原因说明。

### 4.4 类型变更与兼容性

- 纯类型重构不得改变运行时行为。
- 若类型变更收紧了 API 契约，需在同一次变更中同步更新测试与文档。

---

## 5. 代码规范检查（Ruff）

### 5.1 基线配置

Ruff 以 `pyproject.toml` 中配置为准，当前包括：
- `line-length = 88`
- 规则族：`E`、`W`、`F`、`I`、`B`、`C4`、`UP`
- `ignore` 与 `per-file-ignores` 中的既有例外

### 5.2 检查范围与执行

- 对本次变更涉及的 `src/` 与 `demos/` 文件，lint 必须通过。
- 导入顺序必须符合 Ruff/isort 规则。
- 修复 lint 问题时应优先提升可读性与正确性，而非扩大抑制范围。

### 5.3 允许的例外

- 可沿用 `pyproject.toml` 中已有、已说明理由的例外。
- 新增忽略规则必须最小化范围，并在变更说明中写明理由。

### 5.4 禁止行为

- 不得为图省事关闭整类规则。
- 不得提交死代码、不可达分支或未使用导入。

---

## 6. Pull Request 合并检查清单

仅当以下条件全部满足时，变更才可合并：
- 单元测试通过（`pytest`）。
- 类型检查通过（`pyright`）。
- 代码规范检查通过（`ruff check src demos`）。
- 新增或修改行为已有测试覆盖。
- 新增公共 API 的类型与文档与 `docstring_spec_cn.md` 保持一致。
- 若由 AI Agent 产出，提交说明包含“变更摘要 + 验证结果 + 残留风险（如有）”。

---

## 7. 失败时的修复顺序建议

检查失败时，建议按以下顺序处理：
1. 先修复功能正确性与失败测试。
2. 再修复反映真实契约问题的类型错误。
3. 最后处理 lint 与导入/顺序问题。
4. 提交前重新完整执行全部检查命令。

该顺序可降低返工成本，避免用样式改动掩盖行为回归问题。

---

## 8. 最小提交流程（适用于人工与 AI Agent）

每次变更按以下最小流程执行：
1. 先实现单一目标改动（避免混合多个主题）。
2. 运行 `pytest`、`pyright`、`ruff check src demos`。
3. 若失败，按第 7 节顺序修复并重跑。
4. 更新相关文档与注释（尤其是公开 API 的 docstring）。
5. 在提交说明中记录：
	- 本次改了什么。
	- 如何验证通过。
	- 是否有已知限制或后续事项。
